# SecondBrain — обязательные правила разработки

Этот файл обязателен для всех AI-агентов, работающих с репозиторием.

## Главная цель

Разрабатывать персональную систему управления знаниями SecondBrain строго по проектной документации. Качество и сопровождаемость важнее скорости.

## Источники истины

При конфликте применять приоритет:

1. `SecondBrain_Development_Checklist.md`
2. документация в `/docs`
3. инструкции владельца проекта

Если требуется изменить направление проекта, сначала нужно изменить чек-лист в установленном порядке.

## Абсолютный запрет

До завершения всего раздела **«1. Проектирование»** запрещены программный код, база данных, Telegram-бот, UI, реализация, а также анализ существующего кода, зависимостей и схемы БД. Разрешена только работа с документацией.

Раздел «Проектирование» считается завершённым только после того, как выполнены все его пункты, Codex проверил полноту и непротиворечивость документации, а документация стала достаточной для реализации без дополнительных догадок.

## Порядок работы

Всегда следовать чек-листу сверху вниз.

- Не пропускать пункты.
- Не работать одновременно над несколькими незавершёнными пунктами.
- Завершить одно решение.
- Обновить документацию.
- Обновить чек-лист, если решение завершает его пункт.
- Дополнить историю.
- Создать коммит.
- Перейти к следующему решению.

Если один пункт требует нескольких архитектурных решений, каждое решение оформляется отдельным коммитом. Пункт отмечается выполненным только после завершения всех входящих решений.

## Коммиты

Codex самостоятельно создаёт коммиты без предварительного согласования изменений с владельцем. Каждое отдельное решение получает отдельный Git-коммит. Не объединять несвязанные изменения.

Допустимые префиксы: `docs:`, `feat:`, `fix:`, `refactor:`.

## История

Постоянная история хранится в `/docs/history/YYYY-MM.md`.

- Только добавлять новые записи.
- Существующие записи не удалять и не переписывать.
- Запись истории включать в тот же коммит, что и соответствующее решение.
- В поле хеша писать `текущий коммит`, поскольку коммит не может содержать собственный хеш.
- Точный хеш указывать в итоговом отчёте.

Каждая запись содержит метку времени, выполненную работу, принятые решения, изменённые файлы, поле коммита и следующую задачу.

## Проектная память

Перед началом любой работы обязательно прочитать:

1. `SecondBrain_Development_Checklist.md`;
2. все файлы в `/docs/history/`;
3. остальную документацию в `/docs/`.

Документация репозитория — единственная надёжная память проекта.

## Документация

- Документацию обновлять до реализации.
- Каждое архитектурное решение документировать.
- Не вводить недокументированное поведение.
- Документационный пункт готов, когда он полный, непротиворечивый и достаточен для реализации без догадок.

## Архитектурные принципы

- Сохранять архитектуру простой.
- Избегать лишних абстракций и переусложнения.
- Предпочитать читаемость изобретательности.
- Использовать явный код вместо магии.

## Технологический стек

Если документация не изменит решение: Python 3.12+, SQLite, python-telegram-bot, SQLAlchemy, `.env`, pip и venv.

## База данных

До программирования финализировать схему. Документировать каждую таблицу, поле, связь и ограничение.

## Данные и внешняя обработка

- Сохранять исходные данные пользователя.
- Сохранять утверждённые результаты распознавания речи и получения метаданных ссылок.
- Никогда не перезаписывать необработанные данные.
- Исходное аудио хранить, если документация явно не установит иное.

## Стиль кода

- Использовать аннотации типов.
- Предпочитать небольшие модули и функции.
- Давать сущностям понятные имена.
- Не дублировать код.
- Избегать глобального состояния.

## Логирование

Использовать стандартный модуль Python `logging`. Не использовать `print()` в production-коде.

## Критерий готовности реализации

Задача реализации завершена только тогда, когда реализация работает, документация обновлена и чек-лист обновлён.

## Запрещено

- Придумывать функции без документированного решения.
- Пропускать пункты чек-листа.
- Создавать заглушки реализации.
- Менять архитектуру без предварительного документирования.
- Начинать реализацию до завершения проектирования.

## Общение

Общаться с владельцем проекта только на русском языке. Код и идентификаторы могут оставаться на английском.

Метка `[YYYY-MM-DD HH:MM]` обязательна в итоговых отчётах о завершённых задачах. Вопросы и промежуточные сообщения могут быть без метки.

После каждой завершённой задачи итоговый отчёт содержит:

- Completed;
- Changed;
- Commit;
- Next.

## Итоговый принцип

Одно решение. Один коммит. Строгий порядок чек-листа.
