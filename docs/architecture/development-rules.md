# Правила разработки

## Область действия

Этот документ дополняет обязательный `AGENTS.md`. При противоречии действует приоритет, установленный в `AGENTS.md`: чек-лист, документация, затем текущие инструкции владельца.

## Рабочий процесс

1. Перед любой задачей прочитать чек-лист, историю и актуальную документацию.
2. Работать только над первым незавершённым пунктом чек-листа.
3. До реализации зафиксировать поведение и архитектурное решение в документации.
4. Реализовать минимальный объём, полностью закрывающий одно решение.
5. Добавить или обновить проверки, соответствующие изменению.
6. Выполнить доступные проверки качества.
7. Обновить документацию, чек-лист и историю.
8. Создать один самостоятельный коммит.

До завершения и итоговой проверки всего раздела «Проектирование» шаги реализации и анализа кода запрещены правилами `AGENTS.md`.

## Git

- Основная рабочая ветка персонального проекта — `main`.
- Обычные изменения коммитятся непосредственно в `main`.
- Отдельные ветки и pull request не обязательны; они могут использоваться для рискованных экспериментов или внешнего участия только по отдельному решению.
- Один коммит содержит одно решение и связанные с ним документацию, тесты и запись истории.
- Не объединять несвязанные исправления.
- Не переписывать опубликованную историю без прямого указания владельца.
- Не коммитить секреты, пользовательские данные, базу, аудио, резервные копии и журналы.

Допустимые префиксы сообщений коммита определены в `AGENTS.md`: `docs:`, `feat:`, `fix:`, `refactor:`.

## Документация и история

- Документация является проектной памятью и обновляется раньше кода.
- Новое поведение без документации запрещено.
- История в `docs/history/YYYY-MM.md` только дополняется; опубликованные записи не редактируются и не удаляются.
- Запись истории входит в тот же коммит, что и решение.
- В записи истории используется `текущий коммит`, точный хеш сообщается в итоговом отчёте.
- Пункт чек-листа отмечается выполненным только после завершения всех входящих решений.

## Python

- Поддерживаемая версия — Python 3.12 или новее в пределах совместимости зависимостей.
- Для публичных функций, методов и значимых структур используются аннотации типов.
- Функции и модули должны оставаться небольшими и иметь одну понятную ответственность.
- Предпочтительны явные зависимости, передаваемые при сборке приложения; скрытое глобальное состояние запрещено.
- Дублирование логики между Telegram-слоем, сервисами и хранилищем запрещено.
- Не использовать `print()` в рабочем коде; применяется стандартный `logging`.
- Ошибки внешних сервисов обрабатываются явно и не должны приводить к потере исходной записи.
- Пользовательские данные не перезаписываются результатами распознавания или AI-анализа.

## Инструменты качества

Настройки инструментов хранятся в `pyproject.toml`:

- `ruff` — форматирование и статические проверки стиля;
- `mypy` — проверка типов прикладного пакета;
- `pytest` — модульные и интеграционные тесты.

Перед коммитом реализации выполняются:

```text
ruff format --check .
ruff check .
mypy src/secondbrain
pytest
```

Если полный набор временно невозможно выполнить из-за недоступной внешней среды, агент выполняет все доступные проверки, не скрывает пропуск и не объявляет задачу полностью проверенной.

Для коммита, изменяющего только Markdown, обязательны проверка `git diff --check`, чтение итогового diff и проверка непротиворечивости затронутых документов. Python-инструменты без изменения Python-кода запускать не требуется.

## Тестирование

- Новое поведение сопровождается тестом на основной сценарий и значимые ошибки.
- Исправление дефекта сначала должно быть воспроизводимо тестом, если это возможно без несоразмерной сложности.
- Модульные тесты не обращаются к реальным Telegram, ChatGPT и сервисам распознавания.
- Интеграционные тесты используют временную базу и подменённые внешние API, кроме отдельно обозначенных ручных проверок.
- Тестовые данные обезличены; реальные сообщения и аудио владельца не используются.
- Время и часовой пояс в тестах фиксируются, особенно для операций в 00:00 и 21:00 по Москве.

## База данных

- Схема и ограничения утверждаются до реализации хранилища.
- Все изменения данных выполняются в явной транзакции.
- Операции, повторно полученные из Telegram или планировщика, должны быть идемпотентными и не создавать дубликаты.
- Разрушающее изменение схемы требует резервной копии и документированного пути восстановления.
- Способ миграций определяется после утверждения финальной схемы БД; до этого инструмент миграций не выбирается.

## Внешние сервисы

- Telegram, распознавание речи и ChatGPT вызываются через отдельные адаптеры или сервисы.
- Для сетевых операций задаются ограниченные тайм-ауты и контролируемые повторы.
- Повтор не должен создавать вторую запись или повторно применять завершённое действие.
- При недоступности внешнего сервиса исходные данные сохраняются, а операция получает состояние для безопасного повтора.
- В журналы не выводятся токены, полный `.env` и иные секреты.

## Безопасность данных

- Секреты хранятся только в локальном `.env` или переменных окружения.
- В Git находится только `.env.example` без значений.
- Доступ к боту разрешён одному явно настроенному Telegram-пользователю.
- Обычное действие «В корзину» только скрывает запись. Корзина хранит максимум 30 записей; при превышении лимита сервис корзины автоматически и без дополнительного подтверждения физически удаляет самые старые записи до достижения лимита.
- Резервные копии проверяются на возможность восстановления.

## Критерий готовности изменения

Изменение готово, когда одновременно выполнены условия:

- утверждённое поведение реализовано полностью;
- исходные данные не теряются и дубликаты не создаются;
- затронутые тесты проходят;
- проверки качества проходят либо явно указан объективный внешний блокер;
- документация, чек-лист и история актуальны;
- рабочее дерево не содержит случайных изменений;
- создан один понятный коммит и в итоговом отчёте указан его точный хеш.
