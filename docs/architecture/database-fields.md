# Поля базы данных

## Общие соглашения

- SQLite хранит идентификаторы как `INTEGER` с автоинкрементом, если для таблицы не указано иное.
- Логические значения хранятся как `BOOLEAN` средствами SQLAlchemy с обязательным ограничением `0/1`.
- Момент времени хранится как строка `TEXT` в ISO 8601 в UTC с точностью до секунд. В коде используются aware-объекты `datetime`; московский часовой пояс применяется только при расчёте расписания и отображении.
- Поля JSON хранятся как `TEXT` с валидным JSON и применяются только для переменного служебного контекста, а не вместо основных колонок.
- Все текстовые значения, отмеченные обязательными, после удаления окружающих пробелов не могут быть пустыми.
- История промежуточных ручных правок текста не хранится. Сохраняются неизменяемый источник и текущий отображаемый текст.

Точные внешние ключи и правила удаления рассматриваются следующим пунктом чек-листа «Связи».

## `records`

| Поле | SQLite | Обязательное | Назначение |
|---|---|---:|---|
| `id` | `INTEGER` | да | Первичный ключ записи. |
| `display_text` | `TEXT` | нет | Текущий текст, показываемый пользователю; `NULL` разрешён только ожидающей распознавания голосовой записи, после успеха текст обязателен. |
| `record_type` | `TEXT` | да | `thought` или `task`; других типов в первой версии нет. |
| `lifecycle_state` | `TEXT` | да | `inbox`, `processed` или `task`; корзина и выполнение представлены отдельными полями. |
| `task_list` | `TEXT` | нет | `today`, `tomorrow` или `week` для любой задачи, включая выполненную, скрытую или находящуюся в корзине; для мысли `NULL`. |
| `completed_at` | `TEXT` | нет | Время последней отметки выполнения; `NULL`, если задача активна или выполнение отменено. |
| `task_active_since` | `TEXT` | нет | Начало непрерывного периода невыполненной задачи; при проверке используется его московская календарная дата, перенос списка поле не сбрасывает. |
| `stale_prompted_at` | `TEXT` | нет | Время единственного вопроса о задаче старше восьми дней в текущем периоде активности; очищается при начале нового периода. |
| `stale_prompt_message_id` | `INTEGER` | нет | Telegram ID текущего неотвеченного вопроса; очищается после ответа, удаления сообщения или любого действия с задачей через обычный список. |
| `hidden_at` | `TEXT` | нет | Время скрытия выполненной задачи ежедневной обработкой. |
| `trashed_at` | `TEXT` | нет | Время перемещения в скрытую корзину. |
| `pre_trash_lifecycle_state` | `TEXT` | нет | Состояние непосредственно перед корзиной для восстановления. |
| `pre_trash_task_list` | `TEXT` | нет | Временной список непосредственно перед корзиной. |
| `pre_trash_completed_at` | `TEXT` | нет | Отметка выполнения непосредственно перед корзиной. |
| `pre_trash_hidden_at` | `TEXT` | нет | Состояние скрытия непосредственно перед корзиной. |
| `created_at` | `TEXT` | да | Время создания записи в БД. |
| `updated_at` | `TEXT` | да | Время последнего изменения текущего представления или состояния. |

Правила совместимости полей:

- только задача может иметь `task_list`, `completed_at`, `task_active_since`, `stale_prompted_at`, `stale_prompt_message_id` или `hidden_at`;
- любая задача имеет ровно один `task_list`, который сохраняется при выполнении, скрытии и помещении в корзину;
- запись с `lifecycle_state = inbox` или `processed` имеет `record_type = thought`, а с `lifecycle_state = task` — `record_type = task`;
- `display_text = NULL` допустим только у записи с голосовым `source_messages`, пока нет успешного `speech_recognition`; все остальные записи имеют непустой `display_text`;
- задача не имеет строк в `record_tags`;
- разобранная нетасковая запись имеет `task_list = NULL`; обычно у неё есть теги, но после удаления последнего тега она остаётся в «Разобранных» без тегов;
- запись в корзине имеет `trashed_at` и заполненные поля, необходимые для восстановления;
- восстановление очищает `trashed_at` и `pre_trash_*` после возврата прежнего состояния;
- отмена выполнения до полуночного скрытия очищает `completed_at` и продолжает исходный непрерывный период `task_active_since`;
- перенос между временными списками и игнорирование вопроса не очищают `stale_prompted_at`; поле очищается только при начале нового периода активности задачи;
- `stale_prompt_message_id` заполнен только пока соответствующий вопрос существует в Telegram; после ответа или удаления при следующей проверке поле очищается;
- возобновление уже скрытой выполненной задачи очищает `completed_at`, `hidden_at`, `stale_prompted_at` и `stale_prompt_message_id`, выбирает новый `task_list` и начинает новый период `task_active_since`.

## `source_messages`

| Поле | SQLite | Обязательное | Назначение |
|---|---|---:|---|
| `id` | `INTEGER` | да | Первичный ключ источника. |
| `record_id` | `INTEGER` | да | Единственная запись, созданная из сообщения. |
| `telegram_chat_id` | `INTEGER` | да | Идентификатор разрешённого чата. |
| `telegram_message_id` | `INTEGER` | да | Идентификатор сообщения внутри чата. |
| `source_kind` | `TEXT` | да | `text` или `voice`. |
| `raw_text` | `TEXT` | нет | Исходный текст Telegram без очистки и правок; для голоса может быть `NULL`. |
| `telegram_file_id` | `TEXT` | нет | Стабильный Telegram file ID голосового сообщения. |
| `telegram_file_unique_id` | `TEXT` | нет | Уникальный Telegram ID файла для обнаружения повторов. |
| `audio_path` | `TEXT` | нет | Относительный путь к временному исходному аудио внутри `data/audio/`; очищается после успешного распознавания. |
| `telegram_sent_at` | `TEXT` | да | Время сообщения по данным Telegram. |
| `received_at` | `TEXT` | да | Время фактического получения обновления ботом. |
| `confirmation_sent_at` | `TEXT` | нет | Время успешной отправки единственного итогового подтверждения; `NULL`, пока запись ждёт своей очереди или распознавания. |
| `created_at` | `TEXT` | да | Время сохранения источника в БД. |

Для `text` обязателен `raw_text`, а голосовые поля равны `NULL`. Для `voice` обязательны оба Telegram ID файла. `audio_path` обязателен, пока нет успешного результата распознавания, и равен `NULL` после удаления успешно обработанного файла. Это единственное изменяемое служебное поле неизменяемого источника.

Пара `telegram_chat_id + telegram_message_id` уникальна и предотвращает повторное создание записи при повторной доставке обновления. `record_id` также уникален: в первой версии одна запись создаётся ровно из одного сообщения Telegram.

Итоговые подтверждения отправляются только для строк с `confirmation_sent_at IS NULL` в порядке `telegram_sent_at`, затем `telegram_message_id`. Поле заполняется только после успешной отправки ответа Telegram; повторный запуск продолжает с первой неподтверждённой строки и не отвечает повторно на уже отмеченные сообщения. Если процесс аварийно завершился строго между успешной отправкой Telegram и записью времени, подтверждение может повториться, но вторая пользовательская запись не создаётся.

## `tags`

| Поле | SQLite | Обязательное | Назначение |
|---|---|---:|---|
| `id` | `INTEGER` | да | Первичный ключ тега. |
| `name` | `TEXT` | да | Отображаемое название. |
| `normalized_name` | `TEXT` | да | Нормализованное имя для проверки уникальности без учёта регистра и окружающих пробелов. |
| `is_system` | `BOOLEAN` | да | Признак начального тега, поставляемого системой. |
| `sort_order` | `INTEGER` | да | Порядок показа; пользовательские теги располагаются перед «Прочее». |
| `created_at` | `TEXT` | да | Время создания. |
| `updated_at` | `TEXT` | да | Время последнего переименования или изменения порядка. |

`normalized_name` уникально. Пустое имя запрещено. Длина, нормализация, переименование и удаление определены в `docs/architecture/tags.md`.

## `record_tags`

| Поле | SQLite | Обязательное | Назначение |
|---|---|---:|---|
| `record_id` | `INTEGER` | да | Разобранная нетасковая запись. |
| `tag_id` | `INTEGER` | да | Назначенный тег. |
| `assigned_at` | `TEXT` | да | Время назначения. |

Составной первичный ключ — `record_id + tag_id`. Повторное назначение той же пары не создаёт новую строку.

## `processing_results`

| Поле | SQLite | Обязательное | Назначение |
|---|---|---:|---|
| `id` | `INTEGER` | да | Первичный ключ попытки. |
| `record_id` | `INTEGER` | да | Обрабатываемая запись. |
| `source_message_id` | `INTEGER` | нет | Источник, если операция относится непосредственно к входному сообщению. |
| `operation` | `TEXT` | да | Вид операции: распознавание речи или получение метаданных ссылки. |
| `status` | `TEXT` | да | `pending`, `running`, `succeeded` или `failed`. |
| `input_text` | `TEXT` | нет | Точный текст, переданный операции, если применимо. |
| `output_text` | `TEXT` | нет | Текстовый результат, например транскрипция. |
| `output_json` | `TEXT` | нет | Валидный структурированный результат внешней обработки, например метаданные ссылки. |
| `error_code` | `TEXT` | нет | Стабильный код ошибки без секрета. |
| `error_message` | `TEXT` | нет | Безопасное диагностическое сообщение. |
| `attempt_no` | `INTEGER` | да | Номер попытки одной логической операции, начиная с 1. |
| `created_at` | `TEXT` | да | Время создания попытки. |
| `started_at` | `TEXT` | нет | Время фактического начала. |
| `finished_at` | `TEXT` | нет | Время успеха или ошибки. |

Успешный результат требует хотя бы одного `output_*`; ошибка требует `error_code`. Повтор создаёт новую строку с увеличенным `attempt_no`. Для `speech_recognition` перед повтором удаляется прежняя неудачная строка, а после успеха остаётся только успешная строка; ошибки распознавания не накапливаются.

## `scheduled_runs`

| Поле | SQLite | Обязательное | Назначение |
|---|---|---:|---|
| `id` | `INTEGER` | да | Первичный ключ запуска. |
| `job_name` | `TEXT` | да | Стабильное имя плановой операции. |
| `period_key` | `TEXT` | да | Московская расчётная дата или иной период, для которого выполняется операция. |
| `status` | `TEXT` | да | `running`, `succeeded` или `failed`. |
| `telegram_message_id` | `INTEGER` | нет | ID актуального сообщения, созданного плановой операцией; в первой версии используется вечерним напоминанием и очищается после удаления сообщения. |
| `details_json` | `TEXT` | нет | Итоговые счётчики без пользовательского текста и секретов. |
| `error_code` | `TEXT` | нет | Код ошибки запуска. |
| `started_at` | `TEXT` | да | Время начала. |
| `finished_at` | `TEXT` | нет | Время завершения. |

Пара `job_name + period_key` уникальна. Повтор после ошибки обновляет ту же строку запуска, а не повторно применяет уже завершённые действия.

Для вечерней проверки `telegram_message_id` заполняется после успешной отправки напоминания. При нажатии «Разобрать», опустошении «Входящих» или замене сообщения на следующий день бот удаляет сообщение и очищает поле в той же операции. После перезапуска актуальный непустой ID позволяет удалить прежнее сообщение без создания дубликата.

## `bot_sessions`

| Поле | SQLite | Обязательное | Назначение |
|---|---|---:|---|
| `telegram_user_id` | `INTEGER` | да | Первичный ключ; в первой версии существует не более одной активной пользовательской сессии. |
| `state` | `TEXT` | да | Текущий шаг диалога. |
| `record_id` | `INTEGER` | нет | Запись, к которой относится действие. |
| `context_json` | `TEXT` | да | Минимальный валидный JSON контекста шага. |
| `created_at` | `TEXT` | да | Время начала текущей сессии. |
| `updated_at` | `TEXT` | да | Время последнего шага. |
| `expires_at` | `TEXT` | да | Время безопасного истечения незавершённого диалога. |

Истёкшая сессия удаляется или заменяется новым действием, но не изменяет пользовательскую запись автоматически.

## Индексы, необходимые из состава полей

Окончательный перечень подтверждается после проектирования связей, но обязательны индексы для:

- `records(lifecycle_state, task_list, created_at)`;
- `records(record_type, task_active_since)`;
- `records(trashed_at)`;
- уникального `source_messages(telegram_chat_id, telegram_message_id)`;
- очереди `source_messages(telegram_chat_id, confirmation_sent_at, telegram_sent_at, telegram_message_id)`;
- уникального `tags(normalized_name)`;
- обратного поиска `record_tags(tag_id, record_id)`;
- `processing_results(record_id, operation, created_at)`;
- уникального `scheduled_runs(job_name, period_key)`;
- `bot_sessions(expires_at)`.

## Не хранящиеся данные

- Промежуточные версии ручного редактирования текста не сохраняются.
- Токены Telegram и сервиса распознавания речи в БД не сохраняются.
- Полный секретный запрос или HTTP-заголовки внешнего сервиса не сохраняются.
- Производные счётчики кнопок не хранятся: они вычисляются запросами из актуальных строк.
