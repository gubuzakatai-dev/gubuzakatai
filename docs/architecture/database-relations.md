# Связи базы данных

## Общие правила

- SQLite работает с включённым `PRAGMA foreign_keys = ON` для каждого соединения.
- Все внешние ключи проверяются базой данных, а не только кодом приложения.
- Обычное пользовательское действие удаления не удаляет запись физически, а переводит её в скрытую корзину.
- Корзина хранит не более 30 записей. При превышении лимита самые старые записи корзины удаляются физически вместе с зависимыми строками по описанным каскадам.

## Схема связей

| Родитель | Дочерняя таблица | Кратность | Внешний ключ | При физическом удалении родителя |
|---|---|---|---|---|
| `records` | `source_messages` | один к одному | `source_messages.record_id -> records.id` | `CASCADE` |
| `records` | `record_tags` | один ко многим | `record_tags.record_id -> records.id` | `CASCADE` |
| `tags` | `record_tags` | один ко многим | `record_tags.tag_id -> tags.id` | `CASCADE` |
| `records` | `processing_results` | один ко многим | `processing_results.record_id -> records.id` | `CASCADE` |
| `source_messages` | `processing_results` | один ко многим, необязательно для результата | `processing_results.source_message_id -> source_messages.id` | `SET NULL` |
| `records` | `bot_sessions` | один ко многим во времени, не более одной текущей ссылки на пользовательскую сессию | `bot_sessions.record_id -> records.id` | `SET NULL` |

`scheduled_runs` не имеет внешних ключей: это журнал идемпотентных плановых операций, а не дочерние данные конкретной записи.

## Запись и исходное сообщение

- В первой версии каждая `records` имеет ровно одну `source_messages`.
- `source_messages.record_id` имеет ограничение `UNIQUE`, поэтому одно исходное сообщение не создаёт несколько пользовательских записей.
- Уникальная пара `telegram_chat_id + telegram_message_id` защищает от повторного сохранения одного обновления Telegram.
- Изменение вида или состояния записи при разборе «Входящих» не создаёт новую строку `records` и не меняет её связь с источником.

## Записи и теги

- Между `records` и `tags` действует связь многие ко многим через `record_tags`.
- Составной первичный ключ `record_tags(record_id, tag_id)` исключает повторное назначение одного тега одной записи.
- Строки `record_tags` разрешены только для нетасковых записей в состоянии «Разобранные».
- У задачи строк в `record_tags` нет.
- При удалении тега каскадно удаляются только его связи из `record_tags`; пользовательские записи сохраняются.
- Если удалён последний тег записи, она остаётся в «Разобранных», отображается там без тегов и не возвращается во «Входящие».
- Системные и пользовательские теги разрешено переименовывать и удалять; детальные проверки интерфейса будут определены в правилах тегов.

## Результаты обработки

- У одной записи может быть любое количество `processing_results`: отдельная строка соответствует отдельной попытке операции.
- `processing_results.record_id` обязателен и связывает результат с пользовательской записью.
- `source_message_id` заполняется только тогда, когда операция относится непосредственно к исходному сообщению, например при распознавании голоса.
- Удаление отдельного источника обнуляет необязательную ссылку результата, но не уничтожает сохранённый результат обработки.

## Состояние диалога

- Первичный ключ `bot_sessions.telegram_user_id` обеспечивает не более одной текущей сессии на пользователя.
- `bot_sessions.record_id` необязателен: меню и поиск могут выполняться без выбранной записи.
- Исчезновение выбранной записи не должно делать сессию некорректной, поэтому при физическом удалении записи ссылка становится `NULL`.

## Целостность на уровне приложения

Ограничения, зависящие от нескольких таблиц или состояний, выполняются в одной транзакции:

- преобразование входящей записи в задачу удаляет все ошибочно существующие `record_tags` и устанавливает ровно один временной список;
- преобразование входящей записи в нетасковую разобранную запись устанавливает `task_list = NULL` и назначает подтверждённые теги;
- удаление последнего тега не меняет `lifecycle_state` записи;
- перемещение в корзину не удаляет источник, теги и результаты обработки;
- восстановление из корзины возвращает прежнее состояние и прежние связи.

## Ограничение размера корзины

- После помещения записи в корзину в той же транзакции выбираются записи с `trashed_at IS NOT NULL`.
- Если их больше 30, физически удаляются самые старые по `trashed_at`, а при одинаковом времени — по меньшему `records.id`, пока не останется 30.
- Дополнительное подтверждение окончательного удаления не запрашивается: владелец подтверждает удаление до первоначального помещения записи в корзину.
- Каскады удаляют `source_messages`, `record_tags` и `processing_results`; ссылки текущих `bot_sessions` становятся `NULL`.
- Та же проверка выполняется при запуске приложения, чтобы восстановить лимит после ранее прерванной операции или миграции.
