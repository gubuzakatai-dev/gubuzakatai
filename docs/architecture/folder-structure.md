# Структура папок проекта

## Принцип

Проект использует `src`-структуру: весь Python-код приложения располагается внутри единственного пакета `src/secondbrain/`. Каталоги разделяются по ответственности без глубокой вложенности и без преждевременных абстракций.

```text
│   └── backups/

```

## Назначение каталогов приложения

### `src/secondbrain/bot/`

Telegram-интерфейс: получение обновлений, обработчики команд и сообщений, клавиатуры, переходы между экранами и форматирование ответов. Бизнес-правила и прямые запросы к базе данных здесь не размещаются.

### `src/secondbrain/models/`

Модели предметной области и перечисления: записи, задачи, теги, состояния, результаты распознавания и другие структуры данных. Окончательный состав файлов определяется следующим пунктом проектирования.

### `src/secondbrain/services/`

Прикладные операции: приём записей, классификация, ручной разбор «Входящих», управление задачами, поиск, распознавание голоса, получение метаданных ссылок, ежедневная обработка и напоминания. Telegram-детали и SQL-запросы не смешиваются с этими правилами.

### `src/secondbrain/storage/`

Работа с SQLite: соединение, транзакции, таблицы, репозитории и резервное копирование. Детальная схема базы данных проектируется позже по чек-листу.

### `src/secondbrain/config/`

Загрузка и проверка параметров запуска из переменных окружения и конфигурационных значений. Секреты в коде и Git не хранятся.

## Тесты

- `tests/unit/` — изолированные проверки правил и сервисов без Telegram и реальной базы данных.
- `tests/integration/` — совместная работа хранилища, внешних адаптеров и Telegram-обработчиков.
- `tests/fixtures/` — обезличенные тестовые данные и общие фикстуры.

Структура тестов повторяет ответственность каталогов приложения, но не обязана копировать каждый уровень буквально.

## Документация

- `docs/architecture/` — структура проекта, схема данных и архитектурные решения.
- `docs/concept/` — назначение и пользовательские сценарии.
- `docs/ui/` — экраны и взаимодействие Telegram-бота.
- `docs/history/` — неизменяемая хронология решений по месяцам.

## Локальные рабочие данные

- `data/database/` — рабочая SQLite-база и её служебные файлы.
- `data/audio/` — временные исходные голосовые файлы до успешного распознавания; после успеха файлы удаляются.
- `data/backups/` — локальные резервные копии данных.
- `logs/` — журналы выполнения.

Содержимое `data/` и `logs/` не включается в Git. При необходимости пустые каталоги создаются при развёртывании приложения. Это предотвращает публикацию личных данных, аудио, журналов и рабочей базы.

## Служебные сценарии

`scripts/` предназначен только для явных операций сопровождения: запуск, резервное копирование, восстановление и проверка окружения. Основная логика приложения в этот каталог не выносится.

## Ограничения

- Не создавать каталоги, не имеющие документированной ответственности.
- Не дублировать одну функцию между `bot/`, `services/` и `storage/`.
- Не хранить пользовательские данные, секреты или журналы внутри `src/`.
- Не добавлять дополнительные уровни архитектуры, пока они не потребуются утверждённым поведением.

Этот документ определяет только каталоги. Точный состав файлов рассматривается следующим пунктом чек-листа «Структура файлов».
